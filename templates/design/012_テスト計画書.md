# テスト計画書

## 1. テストの目的と範囲
### 1.1 目的
本テストは、構築したシステムが要件定義書および基本設計書の内容を満たし、実業務での利用に耐えうる品質であることを確認することを目的とする。

### 1.2 テスト範囲 (Scope)
* **対象**: 
    - Webアプリケーション全画面
    - バッチ処理（日次・月次）
    - 外部連携機能（決済、基幹連携）
* **対象外**:
    - 外部システム側の内部処理
    - ネットワークインフラ自体（AWS基盤の動作保証など）

## 2. テストの段階と定義
| 段階 | 略称 | 概要 | 担当 | 実施環境 |
| -- | -- | -- | -- | -- |
| 単体テスト | UT | モジュール単体の動作確認。コードカバレッジC1 80%以上を目標。 | 開発者 | ローカル/CI |
| 結合テスト | IT | モジュール間、サブシステム間の連携確認。API連携やDB整合性を確認。 | 開発者 | Dev/Stg |
| システムテスト | ST | システム全体の通し確認。SLO（性能目標）やセキュリティ要件の達成状況も検証。 | テストチーム | Stg |
| 受入テスト | UAT | ユーザーによる業務シナリオに沿った確認。 | ユーザー | Stg/Prod |

## 4. テスト体制
* **テストマネージャー**: 全体進行管理、品質判定
* **テストリーダー**: ケース作成指示、進捗確認
* **テスター**: テスト実施、バグ起票
* **開発担当**: バグ修正、環境構築

## 5. テスト環境・ツール
* **環境**: ステージング環境（本番相当のデータを用意）
* **管理ツール**: 
    - テスト管理: TestRail / Spreadsheet
    - バグ管理: Jira / GitHub Issues
* **テストデータ**: 
    - 個人情報はマスキング済みの本番データコピー、またはダミー生成データを使用。

## 6. 合格基準・完了条件
* すべてのテストケースが実施済みであること。
* Priority「High」以上のバグが残存していないこと。
* 全体のバグ収束曲線が収束傾向にあること。
